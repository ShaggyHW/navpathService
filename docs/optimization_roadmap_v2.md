# NavPath Optimization Roadmap v2 — Post-Phase-4 Analysis

**Date:** 2026-07-14
**Scope:** entire repo as of commit `f7083e6` (includes the 2026-07-14 bidir budget-accounting fix and budget-exceeded retry). This document covers everything that remains AFTER the Phases 0–4 work recorded in `docs/optimization_audit_4m_scale.md`; nothing listed there as DONE is re-proposed.
**Method:** 12 specialist analysis lenses (A\* core, heuristics/ALT, grid/JPS, bidirectional MM, neighbors/macro, snapshot v9, service runtime, builder, memory/µarch, 4M scaling, accuracy/validation, toolchain) plus a live baseline measurement — 91 raw findings, deduplicated to 63, each adversarially verified against the working tree by an independent agent: **60 confirmed, 1 refuted, 2 uncertain**. 76 agents total. The completeness-critic pass was cut short by a session limit; one verifier (§3.4) was replaced by manual verification of its cited lines.
**Constraint:** path accuracy is preserved. Every item is tagged: **bit-exact** (identical paths+costs), **cost-exact** (optimal cost preserved, equal-cost tie paths may differ), **bounded-suboptimal** (quantified bound, opt-in), or **accuracy-risk** (correctness fix or explicitly flagged trade).

## 1. Correctness first — live bugs found by this audit (fix before optimizing)

These affect served results today. All are small.

### 1.1 BUG: budget/cancel truncation returns unproven paths as `status=Found` — cached, and the retry never fires — `accuracy-risk / small`
When the pop budget (or cancel) fires after a path to the goal has been *discovered* but before optimality is *proven*, both searches return the tentative path as `Found`. Unidirectional: the goal-pop break (`search.rs:321`) precedes the budget break (`search.rs:324-325`), and the post-loop check (`search.rs:377`) only tests `g(goal)==INF`, so `search.rs:395` returns `{found:true, status:Found}` with an unproven cost (bites mainly under small operator `NAVPATH_MAX_POPS`). Bidirectional (the default): MM sets `mu/meet` at first frontier contact (`search.rs:553-557`, `597-601`; immediately at `482-485` for teleport-seeded goals) but proves optimality only at `search.rs:503` — a budget break at `523-524` inside that window falls through to `search.rs:652` and returns `Found` with unproven `mu`. The window is widest on exactly the incident class (long, gated, seeded routes near the cap). Downstream: `routes.rs:1062-1072` caches any `Found` result (despite the comment that truncations "must not stick"), and `budget_retry_pops` (`engine_adapter.rs:97`) keys on `status==BudgetExceeded`, so truncated-found results — the ones carrying possibly-suboptimal costs — bypass the retry and stick in the LRU until snapshot swap. The guard at `search.rs:629-631` is dead code (`meet.is_some()` implies `mu.is_finite()`).

**Fix:** return `status = ended.unwrap_or(Found)` at `search.rs:395` and `652` (keep `found=true` + path); delete the dead guard; retry on `BudgetExceeded` regardless of `found` (keep the truncated path as fallback if the retry also truncates); the cache gate then excludes truncated results automatically. Decide whether clients accept `found=true` + `reason:"budget_exceeded"` or add a `truncated` flag. Regression test: tiny graph, global teleport to goal + cheaper walk route, tiny `max_pops` — assert the walk cost is returned.

### 1.2 BUG: virtual-start searches silently omit all fairy-ring edges — `accuracy-risk / small`
`run_route_with_requirements_virtual_start` (`engine_adapter.rs:499-580`) builds its `EngineView` with `extra: ExtraEdges::default()` (`:535-541`) and never populates `fairy_sources/fairy_dests` — the normal path wires them at `:415-432`/`:455-456`. With empty `fairy_sources` the `is_empty()` short-circuit (`search.rs:338-345`) makes fairy hops structurally impossible for every off-graph-start request; routes needing teleport-entry → walk → fairy-hop come back longer, and a goal reachable from teleport entries only via fairy could return a false `NotFound`. The fairy state is already cloned into the same `spawn_blocking` closure (`routes.rs:976-977`) — it just isn't passed.

**Fix (~15 lines):** replicate the eligible-fairy scan in the virtual-start function and set `view.extra.*`. ALT tables already bake the full fairy clique (`search.rs:262-266`), so admissibility is untouched; every fixed cost is ≤ the current one. Add a unit test seeding `astar_multi` with fairy extras.

### 1.3 BUG: route cache keys eligibility by a 64-bit `DefaultHasher` digest, not the mask bits — `accuracy-risk / small`
`RouteCacheKey.mask_hash: u64` (`lib.rs:20`) is a SipHash digest of `mask.satisfied` (`routes.rs:943-948`). Two masks colliding on 64 bits (with equal sid/gid/quick_tele/seed) share a slot, and a hit skips the search entirely — serving a route computed under a different profile; the payload builder's fallback (`routes.rs:695-703`) can then emit metadata for edges the profile doesn't allow. Chance probability is ~2⁻⁶⁵-scale and adversarial exploitation impractical — this is key-hygiene, but the fix is free and produces exactly the packed-mask key §5.4 needs.

**Fix:** store the exact bits — pack `mask.satisfied` (129 tags today) into `[u64; 3]` in the key. Key grows 16 B; lookup cost unchanged.

### 1.4 BUG: producer regression — `tiles_regions` gone from the DB and `idx_tiles_walkable` re-created — `bit-exact / small`
The checked-in `worldReachableTiles.db` has **no `tiles_regions` table** and a re-created `idx_tiles_walkable` (exact duplicate of the PK autoindex). Git archaeology: `fdffdfc` (2026-07-06) had regions/no index at 43.0 MB; `070676d` (2026-07-08) onward regressed; today 57.8 MB with tiles stored three times (table 22.3 + autoindex 18.5 + index 16.9 MB). The builder's region fast path (`load_sqlite.rs:34-75`) is dead and it falls back **silently** to the row-per-tile scan (no log line anywhere in the file).

**Fix:** re-run `migrate_tiles_regions.py` after every export (and `DROP INDEX idx_tiles_walkable`) until the producer emits `tiles_regions` natively and drops the row-per-tile table (at 4M: ~10-15 MB region-native vs 200+ MB — what keeps the DB git-committable). Builder hardening: `warn!` on fallback, and promote the release-compiled-out `debug_assert` on strictly-ascending packed coords (`main.rs:317`) to a hard bail — a malformed region row would otherwise silently corrupt the snapshot's binary-search coordinate lookup.

### 1.5 Seeded budget-retry silently rewrites the request contract — `bounded-suboptimal / small`
The retry answers a seeded request with the deterministic unseeded optimum, with no response marker (`reason` is only set from the final status, `routes.rs:1079-1083`), and caches it under the seeded key — different seeds converge to byte-identical answers. Deliberate, but invisible to clients. **Fix:** (1) surface it (`reason:"seed_dropped"` or a `degraded` field); (2) retry **with the seed at the escalated cap first** (jitter-inflated pops are a small factor; the 4× cap almost certainly succeeds while honoring the seed), unseeded only as second fallback — both attempts already share the deadline/cancel ceiling; (3) this mostly evaporates if §3.4 lands (plateau shrinks → retries become rare).

## 2. Engine hot loop — bit-exact mechanical wins

Ordered by measured-confidence × impact/effort. The loop is memory-bound (PGO proved it), so estimates are deliberately conservative; each item is small and they compound.

### 2.1 Has-macro bitmap: kill the guaranteed per-pop probe into the 9 MB offsets array — `small, ~3-7% now / 5-10% at 4M`
Every expansion calls `macro_neighbors` (`search.rs:368/567/613`) which loads `offsets[u]/offsets[u+1]` from a `Vec<usize>` of nodes+1 entries (9 MB today, 32 MB at 4M; ×2 for the reversed provider) — yet only ~869 of 1.12M nodes have macro edges, so ≥99.9% of probes return an empty range. It is the only removable nodes-scale random access left in the loop. **Fix:** `has_src: Vec<u64>` bitmap (140 KB, L2-resident) or a sorted 869-entry src array (3.5 KB, L1) guarding the macro loop at all three sites; also skips per-pop `filter_map` iterator construction.

### 2.2 Compact macro adjacency: src-indexed instead of dense offsets — `medium, memory win`
Complements 2.1: replace the two dense `Vec<usize>` offset arrays with `srcs: Vec<u32>` (~869 sorted) + `off: Vec<u32>` — −18 MB persistent heap now, −64 MB at 4M, and deletes 2×18 MB of transient build arrays from every load/`/admin/reload`. Slot order preserved → `MacroFilter` indices unchanged, bit-exact. Pair with the bitmap so the per-hit binary search is off the common path.

### 2.3 Software prefetch: neighbor NodeStates + next heap top — `small, ~3-8% now / 10-20% at 4M (measure)`
The relax closure's `ctx.get_g(v)` (`search.rs:355`) is a random 16 B load feeding a data-dependent branch; misses serialize behind mispredicts. Before the relax loop, issue `_mm_prefetch` for the up-to-8 dst NodeStates (the dst slice is already loaded); after `pop()`, peek and prefetch `nodes[peek.id]` for the next stale-check. A third variant — prefetching the ALT row per dst — is ~70-85% wasted bandwidth (only first-discovery neighbors gather) and must be measured separately. `cfg(target_arch = "x86_64")`-gated; bit-exact regardless of outcome.

### 2.4 Single packed-integer heap comparator; optional 4-ary heap — `small, 2-6%`
`Key::cmp` (`search.rs:79-90`) runs two NaN-tolerant f32 chains per sift step; the g-chain runs frequently because quantized ALT creates wide f-plateaus. All keys are provably non-negative finite (every push is `h.is_finite()`-guarded; g sums non-negative weights) → `to_bits` is order-preserving. Compare `k = ((!f.to_bits() as u64) << 32) | g.to_bits()` once (larger k = smaller f, then larger g — exactly today's order); id tie-break only on k-equality. Bit-exact: (g,id) pairs per heap are unique (pushes require strictly smaller g), so the pop sequence is identical. Optional second step, benchmark-gated: 4-ary implicit heap with precomputed-k 16 B entries (4 children per cache line, half the levels; +33% heap bytes). **Bucket/calendar queues were analyzed and rejected**: g does not stay on a coarse lattice (300 vs 300·√2, arbitrary macro weights, jitter) and quantized ALT h is inconsistent — Dial exactness fails twice; do not revisit.

### 2.5 Incumbent-bound push pruning — `small, 1-4% typical, 3-10% on bidir/teleport-goal queries`
Both loops push every improving relaxation. Unidirectional: track `incumbent = g(goal)`; skip pushes with `ng + hv > incumbent` (strict). Bidirectional: skip pushes with `pr > mu` (the mu/meet update already precedes the push). Provably pop-sequence-invariant — pruned entries could never pop before termination — so bit-exact including the `pops` counter. Material where mu goes finite early: bidir MM (default path; the 124 seeded teleport dsts are immediate meet candidates) and goals on/near a teleport dst. Value grows with heap size, i.e. at 4M.

### 2.6 Flatten the relax path — `medium, ~1-3% bundled`
Per improving relaxation, four separately bounds-checked accesses to `nodes[v]` (get_g / set_g / set_parent / h_cached, all one cache line — the waste is ALU/bounds-checks); per edge, a re-matched `params.seed` branch and a re-dispatched `WalkIter` enum. Bundle: (1) fetch `&mut ctx.nodes[v]` once and fold the four accessors into one `relax_state` helper; (2) make the search generic over a jitter policy (monomorphize seeded AND unseeded — production is nearly always seeded); (3) match `WalkGraph::Csr` once outside the pop loop; (4) hoist `max_pops.unwrap_or(u32::MAX)`; (5) sort active landmark indices ascending + precompute byte offsets in `select_active` (bit-exact — max over the same set). Only worth doing together with other rewrites of the same loop body (2.3, 2.7).

### 2.7 Internal walk-CSR iteration: per-node diagonal-bit window + table-driven weights — `small, 1-4%`
`WalkIter::Csr::next` (`neighbors.rs:114-126`) pays per edge: enum match, bounds-checked `diag[slot/8]` byte load (a node's ≤8 slots span ≤2 bitmap bytes, re-derived every edge), and a weight branch. Add `WalkGraph::for_each_neighbor(u, f)`: resolve the enum once, load the ≤2 diag bytes into a u16 window, then per edge `w = W[(bits & 1) as usize]; bits >>= 1` with `static W: [f32; 2]`. Same constants → bit-exact. Also the natural substrate for 2.3's prefetch batch.

### 2.8 Fairy membership bitmap — `small, ~0-2%, bundle only`
Fairy-eligible requests binary-search a ≤53-entry list on **every** pop in both directions (`search.rs:338/537/579`). A snapshot-lifetime bitmap of all ring nodes (137 KB; eligibility re-checked on the ~53 hits) answers "no" in one bit test. Bit-exact. Sub-1% standalone — bundle with other loop work.

### 2.9 `alloc_zeroed` SearchContexts — `small, p99 + RSS`
`vec![EMPTY_STATE; nodes]` (`search.rs:115`) memsets a non-zero 16 B pattern, defeating the zero-page specialization and committing 2×18 MB per thread up front (2×64 MB at 4M; ~2-6 ms cold-thread fill today, 10-20 ms at 4M). All-zero bytes are semantically identical to `EMPTY_STATE` (generation guards + `h` already 0.0; verified no accessor bypasses them). Allocate zeroed → lazy commit, RSS proportional to touched corridor. Caveat: THP `[always]` on this host faults 2 MB granules — apply `MADV_NOHUGEPAGE` to contexts if RSS matters more than TLB. Consider defaulting the search semaphore to physical cores (16) rather than SMT (32): searches are DRAM-bound and 32-way doubles context RSS for marginal throughput.

### 2.10 NodeState hot/cold split: `{g,gen}` / `{parent,h}` — `medium, 0-5% now / 5-15% at 4M`
Failed relax attempts (~4-6× more common than improvements), stale-pop checks, and the bidir meet probe read only g+gen — half of each 16 B record. Split into two 8 B arrays: failed relaxes touch one double-density line. A/B a second variant that drops the h cache entirely (cold = parent only, 12 B/node total): h_active re-gathers are ~1.3-2 contiguous row reads per improved node, second usually L2-hot. Integer g was checked and correctly rejected (300·√2 is irrational; jitter grid ~1.5 ns). Gate on criterion; the 4M upper band needs perf-counter validation.

### 2.11 Hugepage-back `lm_tab`; per-section madvise — `small-medium, ~3-10% now / 5-15% at 4M`
`Snapshot::open` applies one blanket `Advice::Random` (`reader.rs:38`). The ALT table (287 MB now, ~1.02 GB at 4M) is random 256 B row gathers on 4 KiB pages — most touches pay an L2-dTLB miss + walk (Zen 4 L2 dTLB covers 12 MB of 4K pages). Two legs: (a) **per-section policy** — `WillNeed`/prefault the ~42 MB of always-hot non-ALT sections (kills post-reload cold-fault spikes, tens of ms); (b) **2 MiB pages for lm_tab** — `MADV_COLLAPSE`/`MADV_HUGEPAGE` on the mapped range (kernel supports read-only file-backed THP; khugepaged lag applies), with the robust fallback `NAVPATH_ALT_HEAP=1` copying lm_tab into an anonymous `MADV_HUGEPAGE` region (~0.1 s load, +table RSS; THP=always gives fault-time hugepages). ~137 huge pages today / ~489 at 4M — fully TLB-resident. Gate on `perf dtlb_load_misses`. Note: file-backed mmaps never get THP automatically, even on this THP=always host.

## 3. Heuristic strength — fewer pops (cost-exact unless noted)

### 3.1 Evaluate ALL landmarks per node, branchless u16 SIMD; retire the 8-of-64 subset — `medium, up to 1.2-2× pops on the weak classes`
At 64 landmarks a node's interleaved row is 256 B = 4 cache lines, and evaluating the 8 *selected* landmarks already touches ~3.6 of them — **max-over-64 costs the same memory traffic as max-over-8**. The subset exists to save compute that SIMD makes free: per query build three full-width operand vectors from the goal row; per node, two `saturating_sub` passes + lane-max + an INF-lane test reproduce today's exact per-landmark semantics (incl. saturation/unreachable rules) over every valid landmark (~12-16 AVX-512 ops on 4 loads). Full-width evaluation is the tightest bound the table can give — it also eliminates the frontier-drift weakness of point selection and strengthens h=INF pruning. Mirror for `h_active_rev` (and swap `select_active_rev` to anchor-outer row-sequential accumulation). **Zero-code pre-validation:** run the corpus with `NAVPATH_ACTIVE_LANDMARKS=64` — if pops drop <5%, only the small compute win remains; skip the SIMD. (Note: the heuristic *microbench* ignores the env var — use the route benches.) Non-power-of-two landmark counts should pad row stride to 32 slots.

### 3.2 Virtual-start landmark selection is degenerate — `small, ~1.5-3× pops on that class`
`astar_multi` passes `origin=None`, so `select_active(goal, goal, k)` scores every landmark **exactly 0** (`heuristics.rs:90-92`) and the tie-break picks the first k by index — no query tailoring at all on precisely the teleport-seeded class where bounds matter most. Add `select_active_seeded(seeds, goal, k)`: score each landmark by the max bound over (the m cheapest) seed dsts toward the goal — ~124×64 u16 row reads once per query (µs; `select_active_rev` already pays exactly this per bidir search). Any goal-finite subset stays admissible. Moot if 3.1 lands (full-row evaluation has no selection step).

### 3.3 Landmark placement: farthest-point + avoid-style refinement — `medium, benchmark-gated`
Selection is pure farthest-point (`landmarks.rs:226-262`), which favors peripheral dead-ends. Keep FP for the first ~16 picks (spread + per-component guarantee), then refine with *avoid*: probe-root Dijkstra, weight nodes by `d(r,v) − current-ALT-bound(r,v)`, add the max-weight subtree leaf. Rebuild-only, no format change. Two honest caps: the 16 ms-quantum experiment proved short/medium pops are plateau-dominated (placement can't help those), and bidir MM softens one-sided weakness — the credible target is the 300-480k-pop gated/long class (plausibly 1.5-3×), and the more reliable payoff is **equal quality at half the count**: 64→32 landmarks halves lm_tab (~0.5 GB saved at 4M). Strictly A/B on the corpus.

### 3.4 Quantization plateau: quantum-bucketed f comparison in `Key::cmp` — `bounded-suboptimal (≤ B ms additive), small, opt-in — ~8-15× on short routes` *(verifier unavailable; core claims manually re-verified)*
h returns exact multiples of 64 ms (`heuristics.rs:148`) while g is an exact f32 sum, so equal-true-cost diamond nodes get f scattered in `(C*−2q, C*]` instead of tying — the high-g tie-break never fires and every diamond node pops (the documented 138→~2.3k short-route regression; seed jitter scatters f the same way, which is why seeded requests suffer most — both production incidents). Finer quanta were *measured* not to help (2273 vs 2352 pops at 16 ms), ε-weighting needs ε≈0.43 to matter (43% bound — useless), and residual side-tables were **refuted** (floor+subtract-one still strictly understates; exact ties never re-form). The only lever left is the comparator: order by `(f / B) as u32` (B = `NAVPATH_TIEBREAK_BUCKET_MS`, suggest 2×quantum = 128 ms), then HIGH g, then id — diamond nodes collapse into ≤2 buckets and the high-g dive works again. Provable additive bound: returned cost ≤ optimum + B (at goal-pop every open entry has f > f_goal − B); MM stop keeps raw f so its early-stop error is also ≤ B. Rollout: default B=0 (bit-exact off); enable for seeded requests first (their baseline already tolerates jitter-scale wiggle of the same order); gate unseeded enablement on the replay harness (§9.1) showing Δcost ≤ B corpus-wide.

## 4. Search strategy — structural pop reductions

### 4.1 Wire the dormant walk-component section into an eligibility-aware reachability precheck — `medium, kills the multi-second 4M worst case`
The `comp` section (2.24 MB, 491 components) is built, stored, and has **zero runtime consumers** (grep: only a reader test). Gated-goal floods survive h=INF pruning (ALT bakes ALL macro edges + fairy clique regardless of per-request eligibility → finite h everywhere) and bidir gives no relief (an exhausted goal-side heap peeks INF and the loop drains the entire forward side). At 1.1M a flood is ~250-500 ms; at 4M it returns `BudgetExceeded` and the retry **re-floods at 4×** — ~1.5-3 s holding a permit and a 4M context pair. Landmark-less pockets are worse (all-UNREACHABLE goal rows → every landmark dropped → h≡0, pure Dijkstra). Precheck per request, before the permit and `spawn_blocking`: condense once per snapshot (comp of each macro src/dst + req tags, fairy comps, global-dst comps), then a 491-bit fixpoint over eligible special edges; if `comp[goal]` unreachable → `NotFound` in <10 µs. **Exact** because eligibility never gates walk edges. This is the single most important scale-robustness item.

### 4.2 Per-side pop counters + a sound per-query bidir/uni policy — `medium, ~1.5-4× on the weak-backward class`
The backward bound anchors on start + ALL eligible globals (`search.rs:438-444`); with ~125 spread anchors the min-aggregates flatten map-wide on permissive profiles, `pr_b → 2g`, and MM forces a reverse-Dijkstra ball of cost-radius ~C*/2 while a strong-h_f unidirectional search would run a corridor. The measured 1.47× average hides this tail, and `SearchResult` carries only a combined `pops` counter. (1) Add `pops_f/pops_b` — zero-risk observability, prerequisite for the rest. (2) At setup compute `hb_goal = h_active_rev(goal)` and `lb_fwd = min(h_f(start), min over seeds (w + h_f(dst)))`; if `hb_goal < β·lb_fwd` (β≈0.5, env), run unidirectional instead — both engines are exact, so the choice is sound. (3) Later: fractional-MM with fixed per-query p. Note: the incident class was plateau+jitter, *not* provably this — ship (1) first and let data size (2).

### 4.3 Stronger bidir stopping conditions (fmin/gmin stops, or NBS) — `large, benchmark-gated`
The only optimality proof is `mu <= min(tf,tb)` over pr-keyed heaps — fmin/gmin are not readable, so the Holte-et-al extended stops (`mu<=fmin_F`, `mu<=fmin_B`, `mu<=gmin_F+gmin_B+ε`) are unavailable and BOTH sides must raise pr past mu. Options: minimally, an extra f-keyed lazy heap per side to enable the fmin stops + mu-aware alternation; or NBS behind `NAVPATH_BIDIR_ALGO` (waiting/ready dual queues, `lb = max(fmin_F, fmin_B, gmin_F+gmin_B+ε)`, ε = 300 per request from the min eligible edge). Cautions: NBS's 2× theorem assumes consistency (quantized ALT is mildly inconsistent — cap is approximate); a previous non-MM variant measured **0.78×** (slower) before MM replaced it, so the ~2× heap-traffic overhead is not hypothetical. Gate per-side on the diff harness + pop counters from 4.2.

### 4.4 Bidirectional multi-source for virtual starts — `medium, ~1.3-1.5× on that class`
Virtual starts run unidirectional `astar_multi` only; the machinery for bidir is already multi-source-shaped (`select_active_rev` takes arbitrary anchors; the global-seed loop IS multi-source seeding; reconstruction handles parent==MAX). Generalize `astar_bidir` to `origin: Option<u32> + seeds` mirroring `search_core`; anchors = the seed set. Prereqs: 1.2 (fairy fix) first; gate with 4.2's policy (virtual-start anchor sets are maximally teleport-heavy); validate exactly like Phase 4 (500+ pair differential).

### 4.5 Scale-aware budgets — `small, prevents incident #3`
The 1.5M default (`engine_adapter.rs:54-63`) is calibrated to the 1.12M-node map. At ~3.87M mega-component the flood logic inverts: floods stop exhausting the heap under budget (→ NotFound) and start returning BudgetExceeded → the retry authorizes a second full flood. Derive: first-attempt = `max(1.5M, nodes/2)`, retry 4× capped near ~2N (bidir's shared counter can legitimately need up to ~2N across both frontiers). Land AFTER 4.1 so bigger budgets can't lengthen unreachable floods. No-op at 1.1M by construction.

### 4.6 Canonical A\* / mask-driven JPS — the staged plan (the deferred Phase-4 flagship, now fully designed)
The concrete design the audit deferred, validated stage by stage against the current tree. Anomaly census reproduced bit-for-bit against the live DB: **4,406** ordered (node,dir) pairs where a diagonal is banned though both cardinal detours exist, **30** forced diagonals (diagonal present, both detours broken). Data point that reshapes the plan: production traffic is ~always seeded, so JPS initially serves unseeded requests, the (unseeded) budget retry, and benches — the seeded default path stays untouched.

- **Stage 1 — load-derived mask grid (bit-exact, small):** derive per-tile 8-direction masks from the CSR in one rayon pass (<100 ms; 1 B/node — no format change needed, the grid is 100% derivable). Key enabler discovered: node ids follow packed (plane,y,x) order and every CSR row is emitted in ascending direction-bit order, so `slot(u,d) = walk_offsets[u] + popcount(mask[u] & ((1<<d)-1))` gives O(1) direction→slot addressing. **Add a builder assert** pinning the row-order invariant (only symmetry is asserted today) and a load-time check for pre-assert snapshots. Build the static forced-stop bitset: macro srcs + 53 fairy nodes + the 4,436 anomaly sites.
- **Stage 2 — canonical successor pruning, no jumping (cost-exact, medium):** per-(node, incoming-direction) successor table (8 B/node, built at load), successors = natural(d) | forced(u,d) per Harabor-Grastien evaluated on the real masks. **Soundness hole found by verification (must implement):** at exactly the 4,406 anomaly sites, mechanical `<=`-pruning makes the two equal-cost cardinal detours prune *each other*, losing the optimal path — an explicit well-founded equal-cost tie-break (e.g. diagonal-first canonical order, so exactly one member of each equal-cost pair survives) is required for the induction to hold. Walk relax attempts drop ~6.7 → ~2 per expansion; also wire into BOTH bidir walk loops (backward is symmetric; MM termination unaffected since per-side optimal g is preserved) or production sees nothing. ~1.5-2× wall on unseeded plateau-heavy short/mediums.
- **Stage 3 — full JPS jumping (cost-exact, large):** interior tiles never touch heap/NodeState/ALT rows; g accumulated edge-by-edge in f32 exactly as `relax()` does (bit-identical costs per tile sequence); stop at goal / forced neighbor / static forced-stop set (superset stopping is trivially sound under eligibility). ALT evaluated only at jump points preserves admissibility verbatim; h=INF guard still applies. Budget: keep `max_pops` charging expansions; add `max_scanned` (~16 M default — a full flood is ≤8n ≈ 8.7M 1-byte mask steps, i.e. floods get *cheaper*) returning the same `BudgetExceeded`. Land unidirectional behind `NAVPATH_CANONICAL` with shadow-compare; route unseeded traffic to JPS only after it beats bidir MM (1.47×) on the corpus. 5-30× pop reduction on open-field walks.
- **Stage 4 — policy (small):** seeded → today's bidir path untouched (bit-exact); unseeded + the always-unseeded budget retry → pruned path (the recurring incident class structurally lands on the fast path). Optional later: segment-local jitter for seeded variety (bounded <0.1 ms × segments), parked until asked.
- **Stage 5 — bidir×JPS (accuracy-risk, large): do NOT build** unless post-Stage-3 benchmarks show bidir MM still winning somewhere. Frontiers can cross on scanned-over tiles with neither side generating a common node → suboptimal mu can satisfy the stop rule. Requires scan-time cross-probes of the other side's NodeState + a written mirror-ordering completeness proof, with pruning disabled at the 4,436 anomaly sites. Expected residual win <1.5× on a minority class.
- **v9 sections (optional insurance, bit-exact):** builder-precomputed successor tables + JPS+-style jump distances (+16 B/node = +64 MB at 4M) only if measured 4M reload/scan cost warrants; keep load-derivation as the v8 fallback.

**Closed negatives from the same lens (do not revisit):** dead-end/corridor tagging — measured 1.72% of nodes peelable + 1.55% corridor, under the impact floor, park it (re-run `scratchpad/peel.py` on 4M data); goal-bounding/geometric containers for macro edges — **proven unsound** under per-request eligibility (removal breaks full-graph boxes, addition breaks walk-only boxes; quick_tele also rewrites weights) and worthless anyway (~869 src nodes, h=INF already prunes).

## 5. Service layer

### 5.1 Observability: currently zero counters — `small, prerequisite-grade`
The only telemetry is one `info!` line (duration/found/cost/length/status). `res.pops` is never exported; retries are invisible (a retried-then-found request logs `Found`); cache hits/puts uncounted; semaphore 503s and deadline 504s emit **no log line at all**; the audit required the LRU ship with a hit-rate metric — it didn't. Add a `Metrics` struct of relaxed `AtomicU64`s (requests, cache_hits/puts, searches, retries, retry_found, budget_exceeded, cancelled, semaphore_rejects; log2 buckets for pops and search_ms), return a `SearchOutcome{res, retried, attempts_pops, search_wall}` from the adapter, split search vs payload timing, expose via `/health` or `/stats`. Every tuning decision below (cache policy, budget sizing, semaphore, 4M planning) is blocked on this.

### 5.2 Route-cache seed policy — `small + a policy decision, bounded-suboptimal if flipped`
The production client always sends (varying) seeds and `seed` is in the key → search hit-rate is ~0 for the dominant traffic; the virtual-start population (sid collapsed to 0) is the highest-hit-rate class and is also sharded away. Ship 5.1 first, measure; then env-flag `NAVPATH_CACHE_IGNORE_SEED` (serve cached to any seed — loses per-seed tie variety, ≤0.033% relative jitter envelope, precedent already exists in the retry). A per-seed sub-cache CANNOT recover hit-rate — only the policy change can; needs client-owner sign-off. Independent bit-exact riders: exact-bits key (1.3), serialize from the Arc'd entry instead of deep-cloning the path Vec on every hit.

### 5.3 Typed response payloads — the unfinished half of audit §4.3 — `medium, ~1-5 ms on long routes, the whole cache-hit path`
`build_route_payload` is still `serde_json::Value` end-to-end: one `json!` per geometry tile, 5-8 allocs per action over BTreeMaps, `optimize_with_surge_dive` re-parses coords out of Values and clones every action, meta subtrees cloned out of Arcs — ~40-70k transient allocations for a 3k-step response. `#[derive(Serialize)]` action enums + `geometry: Vec<[i32;3]>`; serde serializes `Arc<JsonValue>` without cloning; rewrite surge/dive over typed moves with ownership. Key order changes (client parses by key — confirmed); replay-diff old-vs-new as semantically-equal JSON. Biggest relative win: cache hits, where payload build is essentially all remaining CPU.

### 5.4 Per-profile artifact cache — `medium, ~5-15 µs/request, allocator relief`
Every cache-missing request rebuilds MacroFilter ×2 (959-slot scan walking per-edge req Vecs), eligible globals, fairy sets — pure functions of (snapshot, exact mask bits, quick_tele). Small per-snapshot LRU (~32 entries) keyed on the packed mask bits (shared with 1.3) holding `Arc<ProfileArtifacts>`. While there: flatten `MacroEdgeData.reqs` into one CSR to kill per-edge heap Vecs. Matters on µs-scale short routes and at high QPS; measure profile-repeat rate via 5.1 first.

### 5.5 Bounded context checkout pool — `medium, caps RSS + kills cold-thread p99 spikes`
Thread-local context pairs live on tokio's blocking pool (512 max threads, ~10 s idle reap): the semaphore bounds concurrency but not which threads hold 2×18 MB (2×64 MB at 4M); cache hits spawn permit-free blocking tasks that inflate the pool; low-QPS reaping re-pays the fill on fresh threads. Replace with a checkout pool (capacity = permits) in `AppState`; drop-guard return; contexts survive snapshot swaps via `reset(n)`. Hard-caps search-state at permits × pair-size (4 GB at 4M/32 permits vs ~64 GB theoretical). Compose with 2.9 (zeroed alloc) and the audit's Phase-2 note that explicitly deferred this.

### 5.6 Small riders
- **TCP_NODELAY** (audit §4.6, never landed): `axum::serve(...).tcp_nodelay(true)` — one line. Compression (tower-http, ≥16 KB threshold) only if remote clients are confirmed; current evidence says loopback-only, where both are no-ops.
- **Background blake3 verification after snapshot swap** (`bit-exact/small`): the hash tail is written but never verified; an upward bit-flip in lm_tab = inadmissible heuristic = silently suboptimal routes. After `ArcSwap::store`, hash in a background task (`update_rayon`, ~50-100 ms), on mismatch revert to the kept previous Arc + flag in `/health`. Doubles as page warmup.
- **`validate_layout` alignment gap** (`bit-exact/small`): only 6 of 21 sections are alignment-checked; the other 12 typed sections are `from_raw_parts`-cast with only debug_asserts — UB on a corrupt/hand-crafted header (the reader's safety comment falsely claims coverage). ~10 lines, no compatibility cost.

## 6. Snapshot format (v9 candidates — bundle into ONE version bump when ≥2 are justified)

1. **Mask-grid section** (`bit-exact/medium`): 64×64 pages (512 B presence bitmap + 4096 mask bytes) + sorted chunk directory — 3.6 MB now / ~13 MB at 4M. Jump scans read 64 tiles/cache line vs ~2.4 from CSR.
2. **Chunk-major Hilbert renumbering + explicit coord index** (`cost-exact/large, 4M-gated`): frees ids from (plane,y,x) order (today's binary-search lookup depends on it — the exact objection that deferred Hilbert). Regional lm_tab/NodeState spans compact ~1.4× at 1.1M, ~4× at 4M. Realizable win is dTLB/prefetch locality (the snapshot fits page cache on this host); 5-15% at 4M is the optimistic band — strictly A/B-gated on real 4M data. Eytzinger for `find_node` was evaluated and rejected (≤3 µs/request, obsoleted by page-rank lookup). Seeded jitter and landmark tie-breaks change (ids are hash inputs) — costs stay optimal, tie variety differs.
3. **ALT row layout** (with 3.1): pad row stride to power-of-two slots at non-power-of-two landmark counts; evaluate fw/bw sub-rows vs interleaved pairs for SIMD gathers.
4. **Successor/jump-distance sections** — see 4.6 (only if 4M profiling demands).
5. Riders already listed: per-section madvise defaults (2.11), background verify (5.6).

## 7. Builder

1. **Quantize u16 inside the ALT transpose** (`bit-exact/small`): the blocked transpose materializes full f32 node-major lm_fw/lm_bw while landmark-major columns AND both CSRs are still alive — 4·n·lm·4 B coexist (~4.1 GB at 4M/64lm), then `main.rs:598-603` re-reads it all through a *serial* quantize loop with lm_tab allocated BEFORE the drops. Emit interleaved u16 directly from the transpose; delete the f32 tables and the re-quantize pass (−~1 GB peak at 4M; −0.5-1 s). Also quantize each Dijkstra column as produced (fold into the f32 coverage array for selection) → another ~1 GB off peak.
2. **Write sections into a pre-sized output mmap; hash with `update_rayon`** (`bit-exact/medium`): lm_tab currently exists as a 1 GB heap Vec (at 4M) plus page cache during the write, hashed single-threaded. `set_len` + `MmapMut`, transpose writes quanta directly into the file mapping, `update_rayon` the whole map, keep tmp+rename. Primary win is memory (last GB-scale heap buffer gone); ~0.3-0.6 s wall at 4M.

Honors the audit's Appendix-B refutation (walk-only key) by keying on the full input set. ~9-10 s → ~4-5 s today on a hit; ~25-38 s → ~6-10 s at 4M.
4. **Replace the SipHash tuple-key node map** (audit §2.4, never implemented — `bit-exact/small`): `HashMap<(i32,i32,i32),u32>` probed up to 8×/tile (~9M probes, ~32M at 4M, live through the RSS peak). 5. **Parallelize/de-duplicate the serial graph stages** (`bit-exact/medium`): rayon the symmetry assert; compute walk components once and derive ALT-eligibility components by unioning only the ~3.7k special edges over component ids; build the snapshot CSR before the ALT stage and reuse it; `par_sort_unstable_by_key` the tiles (unique keys → deterministic). ~0.7-1.3 s → ~0.3-0.5 s now; ~3-5 s → ~1-1.5 s at 4M; frees ~324 MB (walk COO) early.

## 8. The 4M plan

**Memory math at 4M (this host, 30 GB):** snapshot ~1.2 GB at 64 landmarks (~540 MB at 24); context pairs 128 MB × permits (bounded to 4 GB by 5.5, or halved by 2.10's 12 B variant); realistic budget fine — the audit-era 7 GB-free figure no longer binds, but re-derive at deploy time.

**What must land before 4M regardless of lever choice:** 4.1 (component precheck — the only remaining multi-second worst case), 4.5 (scale-aware budgets), 5.5 (bounded contexts), 2.9, §9 harness re-baselined on 4M data, and §7 builder items (build stays ~30 s instead of minutes).

**Big-lever decision (gated, do not start now):** the audit's HPA\* sketch is superseded — its per-border-segment entrance dedup is NOT cost-exact. The exact formulation: **CRP-style cell overlay with ALL boundary nodes** and within-cell exact tables; walk legs are never gated, so per-request eligibility lives entirely in the overlay's special edges (the special-to-special closure is 4.19 MB today / ~53 MB at 4M). Cost-exact for unseeded requests (up to f32 summation-order ulps — use the path re-coster from §9.1); seeded traffic needs a policy (flat-path fallback or special-edge-only jitter). Realistic worst-case cross-map gated query on a plain bidir overlay Dijkstra: ~10-50 ms at 4M (add a small overlay ALT table or second level to reach single-digit). Walk-CH was assessed and ranked strictly higher-risk (large separators on uniform-cost open grids, 0.7-1.5 GB shortcut section, minutes of preprocessing) — choose it only if the overlay constant proves too slow AND a plane-0 CH prototype shows shortcut growth <3×.
**Gates:** (A) if unreachable/gated queries dominate the 4M tail → the precheck fixed it, stop. (B) if reachable gated+seeded queries exceed ~2M pops or ~1 s p99, or dropping to 24-32 landmarks regresses pops >2× → commit to the CRP overlay as v9 (prototype plane 0, differential-cost via the §9 harness, flat A\* behind a flag). (C) if p99 <~150 ms and worst legit <~1 s → the incremental path is enough; file the overlay as dormant. First measurement on any 4M snapshot: a builder pass counting per-cell boundary nodes (the table term scales with B²).

### Phase E — Stages 1 + 2a DONE; 2b/3 designed and gated (2026-07-15)

- **Stage 1 — canonical substrate, landed (`engine/canonical.rs`).** The per-tile 8-direction mask grid is DERIVED from the v8 walk CSR at load (83 ms measured, 1 B/node, no format change), validating both load-bearing invariants as it goes (same-plane 8-adjacency; ascending direction-bit row order) and failing SOFT to full expansion on pre-invariant snapshots. O(1) direction→slot addressing (`offsets[u] + popcount(mask & ((1<<d)-1))`) means weights derive from the direction bit — no diag-bitmap lookups. The builder now asserts the row-order invariant next to the symmetry assert (parallel, output byte-identical). Direction recovery unpacks coordinates (never raw key deltas — a ±1 key delta can be an x-boundary wrap) and falls back to the full set for non-adjacent/absent parents; adjacent-macro arrivals are priced soundly because every adjacent macro edge costs ≥ the walk diagonal (validated at grid build; violation disables pruning).
- **Stage 2a — strict-domination pruning, landed and default-on (`NAVPATH_CANONICAL=0` kill switch).** Successor table (8 B/node) drops direction `c` out of `u` (reached via `d` from `p`) only when a local detour `p→x→n` or direct `p→n` is STRICTLY cheaper (exact integer step costs 10/14). Soundness is unconditional — a pruned triple on an optimal path would witness a cheaper path — so every optimal path survives verbatim, node-wise optimal distances are preserved, and bidir MM needs zero new theory. Engages automatically for UNSEEDED searches only (including every budget-retry rung — Stage 4's routing policy fell out for free); seeded traffic is untouched.
- **Two soundness traps documented and structurally avoided.** (1) The audit-verified diamond-anomaly trap: mechanical `<=`-pruning at the 4,406 measured sites makes equal-cost detours prune each other (unit fixture proves both detours survive here). (2) **New finding — the parent-race:** a node first reached at EQUAL cost from a non-canonical parent expands with that parent's pruning set, which can drop the canonical continuation that tie-pruning's path-rewriting proof silently assumes; the classic canonical/JPS literature does not resolve this for mask-constrained grids. Strict-only pruning is indifferent to which optimal parent wins the race.
- **Measured (300 real pairs, diff_canonical):** 0 mismatches, uni AND bidir; pops IDENTICAL (strict-only prunes relax attempts, not expansions — as predicted); wall uni ~1.00x, **bidir +14%**. Replay passes BIT-EXACT with pruning on (tie selection unchanged on the whole corpus), payload diff 48/48, invariance green.
- **Stage 2b (equal-cost tie-pruning, the remaining ~1.5x plateau collapse) — parked on a real proof obligation:** resolve the parent-race (candidates: canonical-parent convergence on equal-g relaxations with a re-push discipline, or (node, incoming-direction) search states) before any tie is ever pruned. Post-3.1 full-width landmarks already took most of the plateau, so the expected payoff is smaller than the audit-era estimate — same lesson as 3.4.
- **Stage 3 (JPS jumping) — all substrate now exists** (masks, slot addressing, forced-stop inputs, differential + shadow harnesses); it inherits Stage 2b's proof obligation wherever it prunes ties, so land 2b first or jump with strict-only stopping sets. **Stage 5 remains do-not-build** per the Phase-4 analysis.

**Phase E perf outcome (bench corpus now measures the production default — grid built in setup):** shorts −11…16% across engines, teleport −13%, everything else within tolerance. Two forensic notes worth keeping: (1) one full run showed a 5.9 s median on a gated bench — per-sample data revealed 2.7-39 s iterations, i.e. a transient system-load window, not the algorithm (isolated re-run: 175 ms, faster than the pre-canonical arm in-process); always check `cand/sample.json` before believing an outlier. (2) The gated flood benches drifted +20-40% in both engine ARMS after the grid allocation entered the process — the third layout-mediated drift for this class since Phase B; its gate tolerance is now 30% and the note stands that in-process A/B (`diff_canonical --gated`: canonical 1.30x FASTER over 40 pairs) is the real signal for this class. Baselines blessed.
